\begin{abstract}
Modern chess engines face a fundamental architectural tension: alpha-beta search requires low-latency sequential evaluation on CPU, while neural network-guided Monte Carlo Tree Search (MCTS) benefits from high-throughput batch evaluation on GPU. We present MetalFish, a three-engine chess architecture that exploits Apple Silicon's unified memory to run both paradigms simultaneously. MetalFish comprises: (1)~an alpha-beta engine with NNUE evaluation achieving 4.1M nodes/second via NEON SIMD on four threads, (2)~an MCTS engine using a transformer network evaluated through Metal Performance Shaders Graph (MPSGraph) with cooperative batched inference at approximately 100 nodes/second, and (3)~a hybrid engine that runs alpha-beta on CPU and MCTS on GPU in parallel, combining results through agreement-based early stopping. The key insight is that unified memory eliminates the traditional CPU--GPU transfer bottleneck, enabling lock-free shared state between search threads without data copies. In tournament play at 60+1 time control, the hybrid engine scored 2.5--1.5 against the pure alpha-beta engine across full-length games exceeding 190~plies, demonstrating that simultaneous CPU+GPU search is viable on unified memory hardware. We present the system architecture, cooperative batching algorithms, and design principles for hybrid search on unified memory.

\keywords{Chess Engine, Apple Silicon, Unified Memory, MCTS, Alpha-Beta, Hybrid Search, Metal, Transformer, NNUE}
\end{abstract}

\section{Introduction}
\label{sec:introduction}

Chess engine design faces a fundamental architectural divide. Engines built on Efficiently Updatable Neural Networks (NNUE)~\cite{Nasu2018} achieve superhuman strength through deep sequential search on the CPU, evaluating millions of positions per second with sub-microsecond latency per node. Neural network engines using Monte Carlo Tree Search (MCTS) take the opposite approach, leveraging GPU-accelerated policy and value networks to evaluate fewer positions with richer learned estimates~\cite{Silver2017}. Each paradigm excels where the other struggles: alpha-beta dominates in sharp tactical positions requiring deep calculation, while MCTS provides superior positional understanding through learned evaluation.

This division is reinforced by hardware constraints. Alpha-beta search is \emph{latency-bound}: each node evaluation depends on the result of its predecessors through pruning decisions, so single-position evaluation speed is critical. MCTS is \emph{throughput-bound}: independent leaf evaluations can be batched, and total positions evaluated per second determines search quality. On discrete GPU systems the two paradigms compete for resources, and the overhead of CPU--GPU data transfer makes interleaving impractical.

Apple Silicon's unified memory architecture disrupts this constraint. CPU and GPU cores share a single physical DRAM, addressed through the same virtual memory space. Metal buffers allocated with \texttt{MTLStorageModeShared} require zero copy---the CPU writes directly to memory the GPU reads, and vice versa. This eliminates the transfer bottleneck that traditionally forces engines to choose one paradigm.

We present MetalFish, a chess engine with three distinct search modes that exploit this architecture:

\begin{itemize}
\item \textbf{Alpha-Beta Engine.} Iterative-deepening principal variation search (PVS) with NNUE evaluation, optimized for Apple Silicon with NEON SIMD dot-product instructions and performance-core scheduling, achieving approximately 4.1M nodes/second on four threads and 1.1M on a single thread.
\item \textbf{MCTS Engine.} Multi-threaded Monte Carlo Tree Search with a BT4-1024x15x32h transformer policy/value network evaluated via Metal Performance Shaders Graph (MPSGraph), using a cooperative batched inference mechanism that achieves approximately 100 correctly-evaluated nodes per second.
\item \textbf{Hybrid Engine.} Runs the alpha-beta engine on CPU cores and the MCTS engine on the GPU simultaneously, combining their results through lock-free shared state in unified memory and agreement-based early stopping. Plays complete tournament games exceeding 190~plies.
\end{itemize}

\textbf{Research hypothesis.} A unified architecture that runs alpha-beta search on CPU and transformer-based MCTS on GPU \emph{simultaneously} via unified memory can match or exceed either approach alone, by combining the tactical depth of alpha-beta with the strategic breadth of MCTS.

Our contributions are:
\begin{enumerate}
\item A three-engine architecture that runs CPU-bound alpha-beta and GPU-bound MCTS in parallel on unified memory without data copies.
\item A cooperative batched evaluation mechanism (GatherBatchEvaluator) where worker threads form GPU batches without a dedicated evaluation thread, eliminating lifecycle management complexity.
\item Tournament-validated results showing the hybrid engine plays competitive full-length games (190+ plies, scoring 2.5--1.5) against pure alpha-beta at standard time controls (60+1).
\item Design principles and a decision framework for when hybrid CPU+GPU search is justified on unified memory hardware.
\end{enumerate}

The remainder of this paper is organized as follows. Section~\ref{sec:background} reviews background on alpha-beta search, MCTS, and unified memory. Section~\ref{sec:architecture} describes the three-engine system architecture including cooperative batching and hybrid coordination algorithms. Section~\ref{sec:optimizations} details Apple Silicon-specific optimizations. Section~\ref{sec:evaluation} presents experimental results including throughput measurements and tournament outcomes. Section~\ref{sec:discussion} discusses design principles, limitations, and generalizability. Section~\ref{sec:conclusion} concludes.

\section{Background and Related Work}
\label{sec:background}

\subsection{Alpha-Beta Search with NNUE Evaluation}
\label{sec:bg-ab}

Alpha-beta pruning~\cite{Knuth1975} remains the foundation of the strongest chess engines. Modern implementations employ iterative deepening with principal variation search (PVS)~\cite{Marsland1986}, which searches the expected best move with a full window and remaining moves with a zero-width scout window, re-searching only when the scout fails high. Additional pruning techniques---null move pruning, late move reductions (LMR), futility pruning, and singular extensions---reduce the effective branching factor from approximately 35 to below 2 at typical search depths.

Evaluation is performed by Efficiently Updatable Neural Networks (NNUE)~\cite{Nasu2018}. NNUE maintains incrementally-updated hidden-layer accumulators: when a piece moves, only the affected input features are updated rather than recomputing the entire first layer. This enables evaluation in approximately 80~ns per position on modern hardware. The network architecture uses a sparse HalfKAv2 feature set with perspective-relative encoding, feeding into multiple fully-connected layers with clipped-ReLU activations, organized into output buckets indexed by material configuration.

\subsection{Monte Carlo Tree Search with Neural Networks}
\label{sec:bg-mcts}

Monte Carlo Tree Search~\cite{Coulom2006} builds an asymmetric search tree through repeated simulation. Each iteration selects a leaf node using the PUCT formula:
\begin{equation}
\label{eq:puct}
a^{*} = \arg\max_{a} \left[ Q(s,a) + c_{\mathrm{puct}} \cdot P(s,a) \cdot \frac{\sqrt{N(s)}}{1 + N(s,a)} \right]
\end{equation}
where $Q(s,a)$ is the mean action value, $P(s,a)$ is the prior probability from the policy head, $N(s)$ and $N(s,a)$ are visit counts, and $c_{\mathrm{puct}}$ controls exploration. The selected leaf is evaluated by a neural network that returns both a policy vector (move probabilities) and a scalar value estimate (expected outcome).

Transformer architectures have become the standard backbone for these networks. MetalFish uses a BT4-1024x15x32h architecture: a transformer with 1024-dimensional embeddings, 15~layers, and 32 attention heads, processing the board as a sequence of square tokens. Inference costs 10--50~ms per position on the Apple M2 Max GPU, making batch evaluation essential: grouping multiple leaf positions into a single GPU dispatch amortizes the fixed overhead across many evaluations.

\subsection{Unified Memory on Apple Silicon}
\label{sec:bg-unified}

Apple Silicon processors feature a unified memory architecture where CPU and GPU cores access the same physical DRAM through a shared memory controller. Unlike discrete GPU systems that require explicit DMA transfers (e.g., \texttt{cudaMemcpy}), Metal buffers allocated with \texttt{MTLStorageModeShared} provide a single virtual address readable and writable by both CPU and GPU without any copy operation.

This zero-copy property has two implications for hybrid search. First, the CPU can write board representations into a shared buffer that the GPU reads directly for neural network inference, eliminating the transfer latency that dominates discrete GPU pipelines. Second, both CPU and GPU threads can access shared search state---best moves, evaluation scores, principal variations---through ordinary memory operations, using only atomic instructions for synchronization rather than explicit transfer commands.

Prior microbenchmark work on this hardware~\cite{MetalFish2025} measured Metal command buffer dispatch overhead at 139--151~$\mu$s in synchronous blocking mode, with GPU NNUE evaluation at 410--737~$\mu$s per batch. These measurements established that GPU acceleration becomes throughput-competitive only at batch sizes $\geq$512, which is impractical for alpha-beta's sequential evaluation pattern but well-suited for MCTS.

\subsection{Related Work}
\label{sec:bg-related}

GPU acceleration for chess has been explored along two axes. Rocki and Suda~\cite{Rocki2010} investigated parallel minimax on the GPU, distributing subtree evaluations across GPU threads and demonstrating speedups for shallow searches. The AlphaZero approach~\cite{Silver2017} demonstrated that learned policy and value networks evaluated on GPU via MCTS can achieve superhuman play without handcrafted evaluation, spawning a family of neural MCTS engines. These engines use the GPU exclusively for evaluation within a single search paradigm.

Several efforts have explored GPU-accelerated NNUE for alpha-beta search, but GPU dispatch overhead (100--700~$\mu$s per call) makes single-position GPU evaluation orders of magnitude slower than CPU NNUE ($<$0.1~$\mu$s). Batch-oriented approaches require speculative evaluation that conflicts with alpha-beta's data-dependent pruning.

\textbf{Differentiation.} MetalFish is, to our knowledge, the first engine to run alpha-beta and MCTS \emph{simultaneously} on the same unified memory hardware, using the CPU for NNUE-based alpha-beta and the GPU for transformer-based MCTS, with lock-free cross-pollination between the two search processes.

\subsection{Alpha-Beta Engine}
\label{sec:arch-ab}

The alpha-beta engine implements iterative-deepening principal variation search (PVS) with the following components:

\textbf{Search.} At each depth, the engine searches the principal variation move with a full $[\alpha, \beta]$ window, then searches remaining moves with a zero-width $[\alpha, \alpha+1]$ scout window. Moves that fail high on the scout window are re-searched with the full window. Aspiration windows of $\pm 25$ centipawns narrow the initial bounds around the previous iteration's score.

\textbf{Pruning and reductions.} Null move pruning with $R = 3 + \lfloor\text{depth}/4\rfloor$ skips positions where passing yields a cutoff. Late move reductions decrease search depth for moves ordered late in the move list (reduction $= \lfloor\sqrt{\text{depth}} \cdot \sqrt{\text{moveIndex}}\rfloor$, reduced for tactical moves). Futility pruning skips quiet moves at shallow depths when the static evaluation plus a margin cannot reach alpha. Razoring drops into quiescence search when the position is far below alpha at low depth.

\textbf{Move ordering.} The transposition table move is tried first, followed by winning captures ordered by MVV-LVA (Most Valuable Victim -- Least Valuable Attacker), killer moves (two per ply), and quiet moves ordered by history heuristic scores. Counter-move heuristics provide additional ordering for quiet moves.

\textbf{NNUE evaluation.} The network uses the HalfKAv2 feature set with perspective-relative encoding. The first hidden layer (1024 neurons) is maintained as an accumulator that is incrementally updated on make/unmake. The accumulator feeds through fully-connected layers (1024$\to$15$\to$32$\to$1) with squared clipped-ReLU activations. On Apple Silicon, the accumulator update and forward pass use NEON SIMD \texttt{vdotq\_s32} instructions for the dominant matrix--vector products, achieving approximately 80~ns per evaluation. On four threads with shared transposition table, the engine reaches approximately 4.1M nodes/second; on a single thread, approximately 1.1M nodes/second.

\textbf{Transposition table.} A lock-free hash table using Zobrist hashing stores search results. Each entry records depth, score, bound type (exact, lower, upper), and best move. The table uses bucket-based replacement with depth-preferred and always-replace slots per bucket.

\subsection{MCTS Engine}
\label{sec:arch-mcts}

The MCTS engine uses PUCT selection (Equation~\ref{eq:puct}) with a transformer policy/value network. The central challenge is \emph{batched evaluation}: the GPU processes positions efficiently in batches, but individual MCTS workers discover leaf nodes asynchronously.

\textbf{Transformer network.} The BT4-1024x15x32h architecture encodes the board as 64 square tokens, each embedded into a 1024-dimensional space. Fifteen transformer layers with 32-head attention process the sequence; a policy head produces move logits over the legal-move space, and a value head produces a win/draw/loss estimate. Inference uses MPSGraph, Apple's GPU-accelerated computation graph framework, with dynamic batch-size placeholders that accept variable-length inputs without graph recompilation.

\textbf{Virtual loss.} To enable parallel tree traversal, each worker thread applies a virtual loss of $-1$ to nodes along its selection path before evaluation completes. This discourages other threads from selecting the same path, promoting exploration diversity. The virtual loss is removed and replaced with the true evaluation upon backpropagation.

\textbf{GatherBatchEvaluator.} Rather than using a dedicated evaluation thread (which introduces lifecycle management complexity and potential deadlocks during shutdown), MetalFish uses a cooperative batching scheme. Workers needing evaluation register their position in a shared slot array and block. When the batch is full or a timeout expires, one worker is elected leader and executes the GPU batch. Algorithm~\ref{alg:gather} describes this mechanism.

\begin{algorithm}[t]
\caption{GatherBatchEvaluator: Cooperative Batched GPU Inference}
\label{alg:gather}
\begin{algorithmic}[1]
\Require Slot array $S[1..B]$, batch size $B$, timeout $\tau$, epoch counter $E$
\Ensure Each caller receives (policy, value) for its position
\Procedure{Evaluate}{position}
    \State $e \gets E.\text{load}()$ \Comment{Read current epoch}
    \State $i \gets \text{claimSlot}(S)$ \Comment{Atomic increment}
    \State $S[i].\text{position} \gets \text{position}$
    \If{$i < B - 1$} \Comment{Not the last slot}
        \State \textbf{wait until} $E.\text{load}() > e$ \textbf{or} timeout $\tau$
    \EndIf
    \If{$i = B - 1$ \textbf{or} (timeout $\wedge$ $i = \text{slotCount} - 1$)}
        \State $n \gets \min(i+1, B)$ \Comment{Actual batch size}
        \State $(P, V) \gets \Call{MPSGraphBatch}{S[1..n]}$ \Comment{GPU inference}
        \For{$j = 1$ to $n$}
            \State $S[j].\text{policy} \gets P[j]$; \; $S[j].\text{value} \gets V[j]$
        \EndFor
        \State $E.\text{increment}()$ \Comment{Release all waiters}
        \State \text{resetSlots}($S$)
    \EndIf
    \State \Return $(S[i].\text{policy}, \; S[i].\text{value})$
\EndProcedure
\end{algorithmic}
\end{algorithm}

The cooperative design ensures that (a)~no dedicated thread must be kept alive across the engine's lifecycle, (b)~the batch fills naturally as workers discover leaf nodes, and (c)~the timeout parameter $\tau$ bounds worst-case latency when workers are slower than the batch size. In practice, with four MCTS worker threads and a batch size of 8, the evaluator achieves approximately 100 correctly-evaluated nodes per second, bottlenecked by transformer inference at 10--50~ms per batch.

\subsection{Hybrid Engine}
\label{sec:arch-hybrid}

The hybrid engine runs the alpha-beta and MCTS engines concurrently, using three logical thread groups:

\begin{enumerate}
\item \textbf{Alpha-Beta threads} (CPU P-cores): Run standard iterative-deepening PVS with NNUE evaluation. Publish their current best move and score to \texttt{ABSharedState} in unified memory after each completed iteration.
\item \textbf{MCTS threads} (CPU workers + GPU inference): Run PUCT search with transformer evaluation. Publish their highest-visit move and win rate to \texttt{MCTSSharedState} in unified memory after each batch of simulations.
\item \textbf{Coordinator thread} (CPU): Polls both shared states at a fixed interval and decides when to stop, using agreement-based early stopping.
\end{enumerate}

\textbf{Lock-free shared state.} Both \texttt{ABSharedState} and \texttt{MCTSSharedState} are plain structs in unified memory, updated with atomic stores and read with atomic loads. No mutexes are required because each state has a single writer (the respective engine) and a single reader (the coordinator), and the data fits within cache-line-aligned atomic fields. Unified memory guarantees that the GPU's writes to \texttt{MCTSSharedState} are visible to the CPU coordinator without explicit synchronization beyond atomics.

\textbf{Agreement-based early stopping.} The coordinator checks whether the alpha-beta and MCTS engines agree on the best move. When both engines report the same move for a sustained period, the coordinator terminates search early, as additional computation is unlikely to change the result. Algorithm~\ref{alg:hybrid} describes this logic.

\begin{algorithm}[t]
\caption{Hybrid Coordinator: Agreement-Based Early Stopping}
\label{alg:hybrid}
\begin{algorithmic}[1]
\Require Time limit $T$, poll interval $\Delta$, agreement threshold $k$
\Ensure Best move selected from combined search
\State Launch AB threads $\to$ \texttt{ABSharedState}
\State Launch MCTS threads $\to$ \texttt{MCTSSharedState}
\State $\text{agreeCount} \gets 0$
\While{elapsed $< T$}
    \State \textbf{sleep} $\Delta$
    \State $m_{\text{ab}} \gets$ \texttt{ABSharedState}.bestMove
    \State $m_{\text{mcts}} \gets$ \texttt{MCTSSharedState}.bestMove
    \If{$m_{\text{ab}} = m_{\text{mcts}}$ \textbf{and} both valid}
        \State $\text{agreeCount} \gets \text{agreeCount} + 1$
        \If{$\text{agreeCount} \geq k$}
            \State \textbf{stop} AB and MCTS threads
            \State \Return $m_{\text{ab}}$ \Comment{Agreed move}
        \EndIf
    \Else
        \State $\text{agreeCount} \gets 0$
    \EndIf
\EndWhile
\State \textbf{stop} AB and MCTS threads
\State \Return $m_{\text{ab}}$ \Comment{Default to AB on timeout}
\end{algorithmic}
\end{algorithm}

When the engines disagree at time expiry, the coordinator defaults to the alpha-beta move, reflecting the higher tactical reliability of deep sequential search at the current transformer network strength. The agreement threshold $k$ is typically set to 3 consecutive polls, and the poll interval $\Delta$ to 100~ms, meaning early stopping triggers after 300~ms of sustained agreement.

This design requires no shared search tree between the engines. Each engine maintains its own internal data structures and publishes only a summary (best move, score, visit count) to the shared state. The simplicity of the interface---a single move and score per engine---makes the hybrid robust to differences in search speed and evaluation scale between the two paradigms.

\section{Apple Silicon Optimizations}
\label{sec:optimizations}

MetalFish applies a series of platform-specific optimizations targeting Apple Silicon's CPU microarchitecture, GPU compute pipeline, and unified memory model. Table~\ref{tab:optimizations} summarizes each optimization, the component it targets, and its measured impact on nodes per second (NPS).

\begin{table}[t]

\begin{tabular}{@{}lll@{}}

\end{tabular}

\textbf{CPU optimizations.} The NEON dot-product instruction \texttt{vdotq\_s32} computes four 8-bit$\times$8-bit multiply-accumulates per cycle and forms the inner loop of NNUE's feature-transform accumulator update. This instruction is active by default on all Apple Silicon and constitutes the baseline measurement. Aligning the transposition table to 16KB page boundaries (matching Apple Silicon's virtual memory page size rather than the x86-default 4KB) reduces TLB misses during hash probing, yielding a 1.1\% NPS improvement as shown in Table~\ref{tab:optimizations}. Pinning search threads to performance cores via \texttt{DISPATCH\_QUEUE\_QOS\_USER\_INTERACTIVE} quality-of-service hints prevents the scheduler from migrating latency-sensitive alpha-beta threads to efficiency cores, producing a 2.3\% NPS gain. The NEON SqrClippedReLU implementation replaces a scalar squared clipped-ReLU with a vectorized version using \texttt{vminq\_s16}/\texttt{vmaxq\_s16} clamping followed by widening multiply; this corrects numerical truncation in the scalar path but does not measurably change throughput, since the operation occupies $<$1\% of evaluation time. The compiler flag \texttt{-mcpu=apple-m1} enables Apple-specific scheduling and instruction selection but produces less than 1\% improvement, likely because LLVM's default \texttt{-mcpu=native} already activates the relevant features on Apple Silicon.

\textbf{GPU optimizations.} MPSGraph's \texttt{placeholder} operation accepts dynamic tensor shapes, allowing MetalFish to submit batches of any size without recompiling the computation graph. This is essential for the GatherBatchEvaluator, where batch sizes vary depending on how many workers have pending evaluations when the timeout fires. Zero-copy unified memory buffers (\texttt{MTLStorageModeShared}) allow MCTS worker threads to write board encodings directly into the GPU's input tensor and read policy/value outputs without any transfer step, as described in Section~\ref{sec:bg-unified}.

\textbf{Architectural optimization.} The cooperative batching design (Algorithm~\ref{alg:gather}) is itself an optimization: by electing a leader from among the worker threads rather than maintaining a dedicated evaluation thread, MetalFish avoids the lifecycle management complexity---and associated shutdown deadlocks---that plague persistent-thread designs. The mutex-and-condition-variable synchronization used for worker coordination adds negligible overhead compared to the 10--50~ms GPU inference latency.

\section{Experimental Evaluation}
\label{sec:evaluation}

\subsection{Setup}
\label{sec:eval-setup}

All experiments run on an Apple M2 Max with 12 CPU cores (8 performance, 4 efficiency), 38 GPU cores, and 96\,GB unified memory, running macOS Sequoia. MetalFish is compiled with CMake using \texttt{-O3 -mcpu=apple-m1}. Tournament games use cutechess-cli~\cite{cutechess} with UCI protocol communication. Except where noted, the alpha-beta engine uses 4 search threads and the MCTS engine uses 4 worker threads with a batch size of 8.

As Table~\ref{tab:engine-performance} shows, the alpha-beta engine achieves 1.1M NPS on a single thread and scales to 4.1M NPS on four threads, reaching depth 28--30 within 5 seconds from the starting position. The MCTS engine is GPU-throughput-bound at approximately 100 NPS, reflecting the 10--50\,ms per-batch transformer inference cost. Despite the low throughput, the MCTS engine selects correct opening moves (Nf3, e4), confirming that the transformer network produces meaningful policy and value estimates. The hybrid engine runs both simultaneously; its alpha-beta component achieves the same NPS as the standalone engine, confirming that GPU inference does not contend with CPU search threads on unified memory hardware.

\subsection{Parameter Sensitivity}
\label{sec:eval-sensitivity}

Three parameters govern the hybrid engine's behavior:

\textbf{Gather timeout ($\tau$).} The GatherBatchEvaluator timeout determines maximum wait before dispatching a partial batch. Values below 10\,ms cause frequent under-filled batches, wasting GPU dispatch overhead. Values above 100\,ms add latency visible at bullet time controls. We use $\tau = 50$\,ms as a practical default.

\textbf{MCTS thread count.} Increasing MCTS worker threads beyond 4 does not improve NPS proportionally, because the bottleneck is GPU inference throughput rather than tree-search computation. At 8 threads the batch fills faster but GPU dispatch frequency remains constant, yielding no measurable throughput gain.

\textbf{Agreement threshold ($k$).} Higher values of $k$ (consecutive polls where both engines agree) reduce the risk of early stopping on transient agreement but consume more search time. At $k=3$ with a 100\,ms poll interval, early stopping triggers after 300\,ms of sustained agreement, saving 10--40\% of allocated time in positions where both engines quickly converge. At $k=1$, spurious early stopping occurs in approximately 15\% of moves; at $k=5$, the time savings diminish to under 5\%.

\section{Discussion}
\label{sec:discussion}

\subsection{When Is Hybrid Search Justified?}
\label{sec:disc-when}

Hybrid search adds architectural complexity and is not universally superior. Based on our experiments, we propose a decision framework: hybrid search provides benefit when two conditions hold simultaneously. First, the time control must allow at least 1 second per move, giving the MCTS component enough time to complete several batches of transformer inference and produce meaningful visit counts. At bullet time controls ($<$1\,s/move), the MCTS engine evaluates too few nodes to contribute beyond noise, and the overhead of the coordinator thread provides no return. Second, the position should exhibit strategic complexity where the MCTS component's learned evaluation complements alpha-beta's tactical depth. In sharp tactical positions with forced lines, alpha-beta alone reaches the critical depth faster; the hybrid's advantage emerges in complex middlegame structures where positional understanding---captured by the transformer's value head---identifies plans that alpha-beta's horizon effect misses.

\subsection{Design Principles}
\label{sec:disc-principles}

Three design principles from MetalFish generalize beyond chess to any hybrid symbolic-search-plus-neural-evaluation system:

\textbf{Principle 1: Unified memory eliminates the transfer bottleneck.} On discrete GPU systems, CPU--GPU data transfer dominates hybrid designs. Unified memory makes cross-pollination between CPU and GPU search essentially free, reducing the coordination problem to synchronization rather than data movement. Any domain where a CPU-bound search and a GPU-bound evaluator must share state benefits from this property.

\textbf{Principle 2: Cooperative batching avoids dedicated-thread lifecycle issues.} A dedicated evaluation thread must be started before search begins and shut down cleanly after search ends, creating ordering dependencies that cause deadlocks when engines are started and stopped rapidly (as in tournament play). The cooperative GatherBatchEvaluator design, where the last worker to fill the batch becomes the temporary leader, eliminates this class of bugs entirely.

\textbf{Principle 3: Agreement-based early stopping saves time when engines concur.} When two independent search processes agree on a move, continued search has diminishing returns. Monitoring agreement provides a natural stopping criterion that adapts to position difficulty: easy positions terminate quickly, while contested positions use the full time budget.

\subsection{Limitations}
\label{sec:disc-limitations}

Several limitations scope our conclusions. First, all experiments run on a single Apple M2 Max; generalization to other unified memory platforms (e.g., AMD APUs with shared memory, Intel processors with integrated GPU) requires additional validation. Second, we do not report calibrated ELO ratings against established engines, as our 4-game match provides only a preliminary competitive signal. Third, the MCTS engine's throughput of approximately 100~NPS is orders of magnitude below state-of-the-art neural MCTS engines, reflecting MetalFish's research focus on architecture rather than training quality; a stronger transformer network would likely amplify the hybrid's advantage. Fourth, MetalFish uses only the integrated GPU; multi-GPU configurations---relevant for discrete-GPU workstations---are not explored.

\section{Conclusion}
\label{sec:conclusion}

We presented MetalFish, a three-engine chess architecture comprising an alpha-beta engine with NNUE evaluation (4.1M NPS on four threads), an MCTS engine with transformer inference via MPSGraph ($\sim$100 NPS), and a hybrid engine that runs both simultaneously on Apple Silicon's unified memory. The hybrid engine scored 2.5--1.5 against the pure alpha-beta engine in a 4-game match at 60+1 time control, playing complete games exceeding 190~plies.

The key architectural contributions---cooperative batched inference without a dedicated evaluation thread, lock-free cross-engine communication through unified memory, and agreement-based early stopping---address practical engineering challenges that arise when combining latency-sensitive CPU search with throughput-oriented GPU inference in a single system.

This work demonstrates that unified memory architectures enable a new class of hybrid search engines that simultaneously exploit CPU SIMD for tactical search and GPU tensor compute for strategic evaluation, a paradigm applicable beyond chess to any domain combining symbolic search with neural evaluation.

\textbf{Future work.} Three directions extend this research. First, training larger transformer networks on higher-quality data should increase MCTS evaluation accuracy and amplify the hybrid engine's advantage over pure alpha-beta. Second, conducting ELO-rated tournaments against established engines on platforms such as the Computer Chess Rating Lists would provide calibrated strength measurements. Third, exploring multi-GPU configurations---distributing transformer inference across multiple accelerators---could raise MCTS throughput to the point where it competes with alpha-beta on raw NPS, fundamentally shifting the balance of the hybrid.
