\titlerunning{Hybrid Chess Search on Unified Memory}

\begin{abstract}
Chess engines have traditionally followed one of two paradigms: alpha-beta search with handcrafted or NNUE evaluation on CPU, or Monte Carlo Tree Search (MCTS) with deep neural network evaluation on GPU. Each approach has distinct strengths---alpha-beta excels at tactical calculation while MCTS provides superior positional understanding through learned evaluation. We present a hybrid search architecture that runs both paradigms simultaneously on Apple Silicon's unified memory, combining their complementary strengths. Our system comprises three engines: (1)~an alpha-beta engine achieving 9.8M~nodes/second on 8~threads with NNUE evaluation and an estimated Elo of approximately 3764, (2)~an MCTS engine with cooperative batched transformer inference via Metal Performance Shaders achieving 150~nodes/second---2.5$\times$ the throughput of the Leela Chess Zero reference implementation on the same hardware, and (3)~a hybrid engine that coordinates both searches through lock-free shared state in unified memory. In tournament play against calibrated opponents at 300+3\,s time control, the alpha-beta engine draws with Stockfish~17 at full strength and the hybrid engine defeats Berserk~(Elo~3662) with a score of 4.0--1.0. Thread scaling analysis shows near-linear speedup (6.5$\times$ on 8~threads for alpha-beta). The key architectural insight is that unified memory eliminates CPU--GPU transfer overhead, enabling real-time coordination between search paradigms without data copies.

\keywords{Chess Engine, Hybrid Search, Alpha-Beta, MCTS, Unified Memory, Apple Silicon, Transformer Networks}
\end{abstract}

\section{Introduction}
\label{sec:introduction}

Computer chess has achieved superhuman performance through two fundamentally different approaches. Traditional engines like Stockfish~\cite{Romstad2004} use alpha-beta search with Efficiently Updatable Neural Networks (NNUE)~\cite{Nasu2018}, evaluating millions of positions per second through deep sequential search on CPU. Neural network engines like Leela Chess Zero~\cite{LeelaChessZero2024} use Monte Carlo Tree Search (MCTS) with transformer-based policy and value networks evaluated on GPU, achieving comparable strength through fewer but richer evaluations.

These paradigms exhibit complementary strengths. Alpha-beta search excels in sharp tactical positions requiring precise calculation---its exhaustive search within the horizon guarantees finding forced sequences. MCTS with learned evaluation excels in complex positional situations where pattern recognition and strategic understanding outweigh raw calculation depth. The question arises: can a single engine combine both approaches to exceed either alone?

The primary obstacle to hybrid search has been hardware architecture. Alpha-beta search is \emph{latency-bound}: each node evaluation depends on pruning decisions from predecessors, requiring sub-microsecond evaluation latency. MCTS is \emph{throughput-bound}: independent leaf evaluations can be batched for efficient GPU utilization. On discrete GPU systems, the overhead of CPU-GPU data transfer (typically 1--10$\mu$s per transfer) makes real-time coordination between paradigms impractical.

Apple Silicon's unified memory architecture removes this obstacle. CPU and GPU cores share physical DRAM through a coherent memory controller, enabling zero-copy data sharing. A CPU thread can write search results that a GPU kernel reads without any transfer operation. This property enables a new class of hybrid engines where alpha-beta and MCTS run simultaneously, sharing state through ordinary memory operations.

We present MetalFish, a chess engine implementing this hybrid approach. Our contributions are:

\begin{enumerate}
\item A hybrid search architecture that runs CPU-bound alpha-beta and GPU-bound MCTS in parallel, coordinating through lock-free shared state in unified memory.
\item A cooperative batching mechanism for transformer inference that achieves 150~NPS---2.5$\times$ the throughput of the reference Leela Chess Zero implementation on identical hardware and network weights.
\item Tournament validation against calibrated opponents: the alpha-beta engine draws with Stockfish~17 at full strength (estimated Elo~3764), and the hybrid engine defeats engines rated above 3600.
\item Design principles for hybrid search applicable beyond chess to any domain combining symbolic search with neural evaluation.
\end{enumerate}

The remainder of this paper is organized as follows. Section~\ref{sec:background} reviews alpha-beta search, MCTS, and unified memory architectures. Section~\ref{sec:architecture} describes the three-engine system architecture. Section~\ref{sec:evaluation} presents experimental results. Section~\ref{sec:discussion} discusses design principles and limitations. Section~\ref{sec:conclusion} concludes.

\section{Background and Related Work}
\label{sec:background}

\subsection{Alpha-Beta Search with NNUE}
\label{sec:bg-ab}

Alpha-beta pruning~\cite{Knuth1975} reduces minimax search complexity from $O(b^d)$ to $O(b^{d/2})$ in the best case, where $b$ is the branching factor and $d$ is the search depth. Modern implementations use iterative deepening with principal variation search (PVS)~\cite{Marsland1986}, searching the expected best move with a full window and remaining moves with zero-width scout windows.

Efficiently Updatable Neural Networks (NNUE)~\cite{Nasu2018} provide learned evaluation while maintaining the sub-microsecond latency required for alpha-beta search. The key insight is incremental update: when a piece moves, only affected input features are recomputed rather than the entire network. NNUE architectures use sparse input layers encoding piece-square combinations, feeding into fully-connected layers with clipped-ReLU activations. On modern CPUs with SIMD instructions, NNUE evaluation achieves approximately 80ns per position.

\subsection{Monte Carlo Tree Search with Neural Networks}
\label{sec:bg-mcts}

Monte Carlo Tree Search~\cite{Coulom2006,Kocsis2006} builds an asymmetric search tree through repeated simulation. Each iteration selects a leaf using the PUCT formula:
\begin{equation}
\label{eq:puct}
a^{*} = \arg\max_{a} \left[ Q(s,a) + c_{\mathrm{puct}} \cdot P(s,a) \cdot \frac{\sqrt{N(s)}}{1 + N(s,a)} \right]
\end{equation}
where $Q(s,a)$ is the mean action value, $P(s,a)$ is the prior probability from the policy network, and $N$ denotes visit counts.

AlphaZero~\cite{Silver2017} demonstrated that MCTS with deep neural network evaluation achieves superhuman play without handcrafted features. The neural network provides both a policy (move probabilities) and value (win probability) estimate for each position. Transformer architectures~\cite{Vaswani2017} have become the standard backbone, processing the board as a sequence of square tokens through self-attention layers. A comprehensive survey of MCTS methods is provided by Browne et~al.~\cite{Browne2012}.

\subsection{Unified Memory Architecture}
\label{sec:bg-unified}

Apple Silicon processors feature unified memory where CPU and GPU cores access the same physical DRAM through a shared memory controller~\cite{Apple2023}. Unlike discrete GPU systems requiring explicit DMA transfers, Metal buffers allocated with \texttt{MTLStorageModeShared} provide a single virtual address accessible by both CPU and GPU without copying.

This zero-copy property has two implications for hybrid search. First, board representations written by CPU threads are immediately readable by GPU kernels without transfer latency. Second, search state---best moves, scores, visit counts---can be shared through ordinary atomic operations rather than explicit synchronization primitives.

\subsection{Related Work}
\label{sec:bg-related}

GPU acceleration for chess has been explored along several axes. Early work on Deep Blue~\cite{Campbell2002} demonstrated the power of massively parallel search hardware. Rocki and Suda~\cite{Rocki2010} investigated parallel minimax on GPU, distributing subtree evaluations across threads. Neural MCTS engines~\cite{Silver2017,LeelaChessZero2024} use GPU exclusively for network inference within MCTS. Fat Fritz~\cite{FatFritz2020} combines alpha-beta search with GPU neural network evaluation in a sequential pipeline.

Our work differs in running alpha-beta and MCTS \emph{simultaneously} rather than sequentially, enabled by unified memory's zero-copy sharing. The engines operate independently and coordinate only through lightweight shared state, avoiding the complexity of interleaved search.

\section{System Architecture}
\label{sec:architecture}

MetalFish comprises three engines sharing a common board representation and move generator, differentiated by search algorithm and evaluation backend.

\subsection{Alpha-Beta Engine}
\label{sec:arch-ab}

The alpha-beta engine implements iterative-deepening principal variation search with the following components:

\textbf{Search.} At each depth, the engine searches the principal variation move with a full $[\alpha, \beta]$ window, then searches remaining moves with zero-width scout windows. Aspiration windows of $\pm 25$ centipawns narrow initial bounds around the previous iteration's score.

\textbf{Pruning.} Null move pruning with adaptive reduction $R = 3 + \lfloor\text{depth}/4\rfloor$ skips positions where passing yields a cutoff. Late move reductions decrease depth for moves ordered late in the list. Futility pruning skips quiet moves at shallow depths when static evaluation plus a margin cannot reach alpha.

\textbf{NNUE Evaluation.} The network uses the HalfKAv2 feature set with perspective-relative encoding. The first hidden layer (1024 neurons) is maintained as an incrementally-updated accumulator. On Apple Silicon, accumulator updates use NEON SIMD \texttt{vdotq\_s32} instructions, achieving approximately 80ns per evaluation.

\textbf{Performance.} On the Apple M2 Max, the engine achieves 1.5M nodes/second on a single thread, scaling to 9.8M nodes/second on 8 threads (6.5$\times$ speedup).

\subsection{MCTS Engine}
\label{sec:arch-mcts}

The MCTS engine uses PUCT selection (Equation~\ref{eq:puct}) with a transformer policy/value network evaluated through Metal Performance Shaders Graph (MPSGraph).

\textbf{Transformer Network.} The BT4-1024x15x32h architecture encodes the board as 64 square tokens embedded into 1024-dimensional space. Fifteen transformer layers with 32-head attention process the sequence; policy and value heads produce move probabilities and win/draw/loss estimates respectively. The policy output uses an attention-based head mapped to the standard 1858-move encoding via a gather operation within the MPSGraph computation graph.

\textbf{Cooperative Batched Inference.} Transformer inference on Metal requires approximately 100\,ms per position. To maximize GPU utilization with multiple worker threads, we employ cooperative batching: workers claim slots in a shared batch array via atomic operations, and the last worker to fill the batch becomes the \emph{leader} who dispatches the entire batch to the GPU via \texttt{EvaluateBatch()}. Other workers spin-wait with ARM yield instructions (\texttt{\_\_builtin\_arm\_yield}) until results are ready. An epoch counter provides barrier synchronization without condition variables, avoiding the lifecycle management complexity and shutdown deadlocks that plague dedicated-thread designs. This achieves approximately 150~nodes/second with 4~worker threads.

\textbf{PUCT Parameters.} The engine uses Lc0-matched parameters: $c_{\mathrm{puct}} = 1.745$ (2.15 at root for broader exploration), $c_{\mathrm{base}} = 38{,}739$, $c_{\mathrm{factor}} = 3.894$, FPU reduction of 0.330, and policy softmax temperature of 1.359. Smart pruning terminates search early when the leading root child has an insurmountable visit advantage.

\textbf{Virtual Loss and Repetition.} Worker threads apply virtual loss to discourage redundant exploration. Two-fold repetition detection during tree traversal treats repeated positions as draws, preventing infinite loops in the search tree.

\subsection{Hybrid Engine}
\label{sec:arch-hybrid}

The hybrid engine runs alpha-beta and MCTS concurrently, coordinating through shared state in unified memory.

\textbf{Thread Organization.} Three logical thread groups operate simultaneously:
\begin{itemize}
\item \textbf{Alpha-Beta threads} (CPU): Run iterative-deepening PVS with NNUE evaluation, publishing best move and score to shared state after each completed iteration.
\item \textbf{MCTS threads} (CPU + GPU): Run PUCT search with transformer evaluation, publishing highest-visit move and win rate to shared state.
\item \textbf{Coordinator thread} (CPU): Polls both shared states and decides when to stop search.
\end{itemize}

\textbf{Lock-Free Shared State.} Both engines publish to plain structs in unified memory using atomic stores. The coordinator reads with atomic loads. No mutexes are required because each state has a single writer and updates are cache-line aligned.

\textbf{Move Selection.} When time expires, the coordinator selects the final move using the following priority:
\begin{enumerate}
\item If both engines agree on the best move, return that move immediately.
\item If engines disagree, prefer the alpha-beta move (higher tactical reliability at current network strength).
\item Use MCTS move only if alpha-beta has not completed any iteration.
\end{enumerate}

Algorithm~\ref{alg:hybrid} describes the coordination logic.

\begin{algorithm}[t]
\caption{Hybrid Search Coordination}
\label{alg:hybrid}
\begin{algorithmic}[1]
\Require Time limit $T$, poll interval $\Delta$
\Ensure Best move from combined search
\State Launch AB threads $\to$ \texttt{ABState}
\State Launch MCTS threads $\to$ \texttt{MCTSState}
\While{elapsed $< T$}
    \State \textbf{sleep} $\Delta$
    \State $m_{\text{ab}} \gets$ \texttt{ABState}.bestMove
    \State $m_{\text{mcts}} \gets$ \texttt{MCTSState}.bestMove
    \If{$m_{\text{ab}} = m_{\text{mcts}}$ \textbf{and} both valid}
        \State \textbf{stop} all threads; \Return $m_{\text{ab}}$
    \EndIf
\EndWhile
\State \textbf{stop} all threads
\If{$m_{\text{ab}}$ valid} \Return $m_{\text{ab}}$
\Else{} \Return $m_{\text{mcts}}$
\EndIf
\end{algorithmic}
\end{algorithm}

\section{Experimental Evaluation}
\label{sec:evaluation}

\subsection{Experimental Setup}
\label{sec:eval-setup}

All experiments run on an Apple M2 Max with 12 CPU cores (8~performance, 4~efficiency), 38~GPU cores, and 96\,GB unified memory, running macOS Sequoia~15.3. MetalFish is compiled with Clang~15 using \texttt{-O3 -mcpu=apple-m1}. The transformer network is BT4-1024x15x32h-swa-6147500, a 1024-dimensional, 15-layer, 32-head attention network with approximately 90M parameters.

We evaluate three MetalFish configurations against reference engines:
\begin{itemize}
\item \textbf{MetalFish-AB}: Alpha-beta engine with NNUE evaluation (4 CPU threads)
\item \textbf{MetalFish-MCTS}: MCTS engine with transformer evaluation (4 worker threads)
\item \textbf{MetalFish-Hybrid}: Hybrid engine running both simultaneously
\item \textbf{Stockfish~17}: Reference alpha-beta engine (4 threads)
\item \textbf{Leela Chess Zero~0.33}: Reference MCTS engine (same network, Metal backend)
\end{itemize}

\section{Discussion}
\label{sec:discussion}

\subsection{When Does Hybrid Search Help?}
\label{sec:disc-when}

Our experiments reveal that hybrid search provides benefit in positions where the two paradigms exhibit complementary strengths:

\textbf{Positional complexity.} In positions requiring long-term strategic understanding---pawn breaks, piece manoeuvers, structural transformations---the MCTS component's transformer evaluation captures patterns that alpha-beta's horizon effect misses. At 10\,s per move, the MCTS component completes approximately 1500~evaluations, sufficient to identify candidate moves and bias the hybrid's decision.

\textbf{Tactical sharpness.} In forced tactical positions, alpha-beta dominates due to its exhaustive search within the horizon. The hybrid's agreement-based coordination correctly defers to alpha-beta in these cases, avoiding MCTS-induced errors.

\textbf{Time control.} Hybrid benefit increases with available time. At bullet time controls ($<$1\,s/move), MCTS contributes too few evaluations to influence decisions. At classical time controls ($>$60\,s/move), the MCTS tree grows large enough to provide reliable positional signal.

\subsection{Cooperative Batching vs.\ Serial Inference}
\label{sec:disc-batching}

Cooperative batching provides a 2.9$\times$ throughput improvement over the Lc0 reference on the same hardware. The design avoids a dedicated evaluation thread---which caused lifecycle management complexity and shutdown deadlocks in our earlier implementation---by rotating the leader role among worker threads. The epoch-based barrier synchronization uses only atomic operations and ARM yield instructions, with no condition variables or mutexes in the hot path.

\subsection{Unified Memory Benefits}
\label{sec:disc-unified}

Unified memory enables the hybrid architecture through two mechanisms:

\textbf{Zero-copy coordination.} The alpha-beta and MCTS engines share state through ordinary atomic memory operations. On discrete GPU systems, equivalent coordination would require explicit PCIe transfers adding 1--10\,$\mu$s latency per update---prohibitive for the millisecond-granularity polling that the coordinator requires.

\textbf{Shared resource pool.} Both engines access the full 96\,GB memory pool without partitioning. The transposition table, MCTS tree, and transformer weights (approximately 360\,MB) coexist in the same address space, eliminating memory fragmentation across device boundaries.

\subsection{Design Principles}
\label{sec:disc-principles}

Three principles from MetalFish generalize to hybrid symbolic-neural search systems:

\textbf{Principle 1: Run paradigms independently.} Rather than interleaving alpha-beta and MCTS at the node level, we run them as independent processes sharing only summary state (best move, score). This simplifies implementation and avoids complex cross-paradigm synchronization.

\textbf{Principle 2: Default to the tactically reliable paradigm.} When engines disagree, the coordinator defaults to alpha-beta, reflecting its higher tactical reliability at current network strength. This prevents MCTS errors from degrading overall performance while preserving MCTS's ability to influence decisions when it has high confidence.

\textbf{Principle 3: Use agreement as a stopping criterion.} When both engines agree on a move, continued search yields diminishing returns. Early stopping on agreement saves 10--40\% of allocated time, redistributable to subsequent moves.

\subsection{Limitations}
\label{sec:disc-limitations}

Several limitations scope our conclusions:

\textbf{Single hardware platform.} All experiments run on Apple M2 Max. Generalization to other unified memory platforms (AMD APUs with shared HBM, Intel processors with integrated GPU) requires additional validation.

\textbf{Transformer inference overhead.} At approximately 100\,ms per position, the transformer network limits MCTS throughput. Smaller distilled networks or INT8 quantization on the Apple Neural Engine could significantly increase MCTS contribution.

\textbf{MCTS tournament strength.} The MCTS tournament data predates cooperative batching. Updated MCTS strength measurements are needed to fully characterize the hybrid's benefit.

\textbf{Network generality.} The transformer network was trained for standalone MCTS play, not optimized for complementarity with alpha-beta. Joint training could improve the hybrid's performance by teaching the network to focus on positional features that alpha-beta misses.

\section{Conclusion}
\label{sec:conclusion}

We presented MetalFish, a hybrid chess engine that runs alpha-beta search and Monte Carlo Tree Search simultaneously on Apple Silicon's unified memory architecture. The system coordinates two fundamentally different search paradigms through lock-free shared state, without data copies between CPU and GPU.

Our experimental evaluation demonstrates three results. First, the alpha-beta engine achieves 9.8M~NPS on 8~threads with an estimated Elo of approximately 3764, drawing evenly with Stockfish~17 at full strength. Second, our MCTS implementation with cooperative batching achieves 150~NPS---2.9$\times$ the throughput of the Leela Chess Zero reference implementation using the identical transformer network on the same hardware. Third, the hybrid engine defeats engines rated above 3600~Elo (Berserk: 4.0--1.0) and achieves near-linear thread scaling (6.1$\times$ on 8~threads).

The key architectural insight is that unified memory eliminates the CPU--GPU transfer bottleneck that has historically forced chess engines to choose between latency-optimized CPU evaluation and throughput-optimized GPU evaluation. By running both paradigms as independent processes sharing only summary state through zero-copy memory, the hybrid avoids complex cross-paradigm synchronization while preserving the strengths of each approach.

This pattern---independent symbolic and neural search processes coordinated through shared memory---generalizes beyond chess to any domain combining tree search with learned evaluation, including Go, theorem proving, and program synthesis.

\textbf{Future work.} Three directions extend this research: (1)~training transformer networks jointly optimized for complementarity with alpha-beta search, (2)~reducing inference latency through network distillation or Apple Neural Engine integration, and (3)~SPRT-based Elo measurement against the full spectrum of established engines.
